1Preprocessing — input pipelinenotebooks/build_nb02.py · backend/fault_injector.py

What the data goes through before either model ever sees it. Seven stages, all wrapped in a single script that produces the .npy arrays the trainers consume.

Stage 1 — Raw CSV loading

Per‑engine directories under data/raw/:

  • gengine1/ — train file IDs [10–19, 30–39] (20 files), test IDs [53–65] (13 files)
  • gengine2/ — train IDs [0–15], test IDs [16–21]
  • pengines/ — two pre‑normalized .xlsx files

load_engine_files() reads each CSV with manually supplied column names — the Bosch headers are inconsistent across files, so column names are imposed, not parsed. Each file becomes one DataFrame.

Stage 2 — Standardization (already done at source)

The Bosch CSVs ship pre‑standardized. A peek at raw_000010_measurement_000000.csv shows values like 1.54, -0.92, -0.74 — those aren't RPM and ppm, those are z‑scores. So build_nb02.py does not apply a StandardScaler and does not persist (mean, std) per channel. The consequence at the UI layer is that physical units have to be reconstructed with hardcoded calibration constants (the units.js table). No information loss for the model; full information loss for "what was this in real units."

Stage 3 — Schema reconciliation

Different engines have different sensors. The notebook reconciles them:

EngineCols loadedUsed as features (feature_cols)
gengine1 13 (5 inputs + 8 outputs) All 13
gengine2 9 (4 inputs + 5 outputs), NoX renamed to NOx 8 shared with g1: [Speed, Load, Lambda, IgnitionAngle, HC, NOx, TempExhaust, TempCatalyst]
pengines 11 tabular (no time‑series) All 11, air_fuel_ratio plays the role of Lambda

Stage 4 — Sliding window construction

Per DataFrame, build overlapping windows of length W = 30 with stride S = 5:

$$X^{(k)}_{t,f} \;=\; \mathbf{a}_{\,s_k + t,\,f},\qquad s_k = k\!\cdot\!S,\;\; t \in [0, W),\;\; f \in [0, F)$$

where a is the engine's (n_rows, n_features) matrix and F = 13 for gengine1. Stride 5 means consecutive windows overlap by 25 timesteps but aren't identical — avoids near‑duplicates inflating sample count. pengines skips this stage because it's static tabular, not time‑series.

Stage 5 — Synthetic fault injection per window

The Bosch dataset has no fault labels — every raw row is "normal operation." So the notebook fabricates labels by perturbing windows. For each window, with probability FAULT_FRAC = 0.25:

  1. Draw fault_type ∈ {1, 2, 3} uniformly.
  2. Add the corresponding offset to the relevant column across all 30 timesteps:
$$\tilde{X}^{(k)}_{t,f^{*}} \;=\; X^{(k)}_{t,f^{*}} \;+\; \Delta_c\qquad \forall t$$

with (f*, Δ_c) chosen by fault class c:

ClassChannel f*Offset Δc
1 (Rich)Lambda−1.5
2 (Lean)Lambda+1.5
3 (Ignition)IgnitionAngle+2.0

Correlated emission offsets Δᵉco, Δᵉhc, Δᵉnox are added to the same window for realism.

Key choices:

  • Perturbation is applied after striding, on a copy of the window. The raw file is never corrupted, and the same row can appear in different windows with different labels.
  • Test sets use fault_fraction = 0.0 — all‑normal raw signal. Synthetic faults for evaluation are reinjected separately by train_bilstm_v2.py at variable magnitudes (the v2 "honest eval" fix in §2).
  • The constant offset is the same value at every timestep — meaning each fault looks like a step change, not a gradual drift. Deliberate simplification.

Resulting class distribution: ~75 % class 0 and ~8.3 % each for classes 1/2/3.

Stage 6 — Per‑engine fault injection variants

pengines injects on air_fuel_ratio (Lambda equivalent) and intake_valve_opening (ignition equivalent) using the same offset magnitudes. Single‑row tensors instead of windows.

2BiLSTM classifiernotebooks/train_bilstm_v2.py

Architecture

$$ \begin{aligned} \mathbf{h}^{(1)} &= \mathrm{BiLSTM}_{96}(X) \in \mathbb{R}^{30\times 192} \\ \mathbf{h}^{(1)} &\leftarrow \mathrm{Dropout}_{0.25}\!\bigl(\mathrm{LN}(\mathbf{h}^{(1)})\bigr) \\ \mathbf{h}^{(2)} &= \mathrm{BiLSTM}_{64}(\mathbf{h}^{(1)}) \in \mathbb{R}^{30\times 128} \\ \mathbf{h}^{(2)} &\leftarrow \mathrm{Dropout}_{0.20}\!\bigl(\mathrm{LN}(\mathbf{h}^{(2)})\bigr) \\ \mathbf{z} &= \mathrm{LN}\!\bigl(\mathbf{h}^{(2)} + \mathrm{MHA}_{h=4,d_k=16}(\mathbf{h}^{(2)},\mathbf{h}^{(2)})\bigr) \\ \bar{\mathbf{z}} &= \tfrac{1}{30}\sum_{t=0}^{29} \mathbf{z}_t \\ \mathbf{u} &= \mathrm{Dropout}_{0.15}\!\bigl(\mathrm{ReLU}(W_1\bar{\mathbf{z}} + b_1)\bigr) \\ \hat{\mathbf{p}} &= \mathrm{softmax}(W_2\mathbf{u} + b_2) \in \Delta^3 \end{aligned} $$

Multi‑head self‑attention

The one non‑obvious block — scaled dot‑product attention applied to the BiLSTM output itself:

$$\mathrm{Attn}(Q,K,V) = \mathrm{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right)V$$ $$\mathrm{MHA}(X,X) = \mathrm{Concat}(\mathrm{head}_1,\dots,\mathrm{head}_4)W^O,\quad \mathrm{head}_i = \mathrm{Attn}(XW^Q_i,\,XW^K_i,\,XW^V_i)$$

Loss — categorical cross‑entropy with label smoothing ε = 0.05

$$y^{\text{LS}}_c = (1-\varepsilon)\,y_c + \frac{\varepsilon}{K},\qquad \mathcal{L} = -\sum_{c=0}^{K-1} y^{\text{LS}}_c \log \hat{p}_c$$

with K = 4. Label smoothing is what stops the softmax saturating at 1.0 (the v1 calibration bug).

Class weighting

sklearn.utils.class_weight.compute_class_weight("balanced", …):

$$w_c = \frac{N}{K\cdot n_c}$$

so the per‑sample loss becomes wyi · Li.

Optimizer & schedule
Adam · lr = 1e-3 · ReduceLROnPlateau(factor=0.5, patience=4) · EarlyStopping(patience=8) · batch_size = 256 · epochs ≤ 60 · val_split = 0.15.

Per‑class threshold calibration

For each class c, sweep t ∈ {0.10, …, 0.99} (200 values) over the eval set and keep the threshold that maximises binary F1:

$$\text{TP}(t) = \sum_i \mathbb{1}[\hat{p}_{i,c}\!\ge\!t \wedge y_i\!=\!c],\quad \text{FP}(t) = \sum_i \mathbb{1}[\hat{p}_{i,c}\!\ge\!t \wedge y_i\!\ne\!c]$$ $$\text{prec}(t) = \frac{\text{TP}}{\text{TP}+\text{FP}},\;\; \text{rec}(t) = \frac{\text{TP}}{\text{TP}+\text{FN}},\;\; F_1(t) = \frac{2\,\text{prec}\!\cdot\!\text{rec}}{\text{prec}+\text{rec}}$$ $$t^{*}_c = \arg\max_{t} F_1(t)$$

These t*c are what the backend later uses for confidence gating.

Honest evaluation set

Faults are injected at variable magnitudes (the v2 fix vs v1's circular eval):

$$\Delta^{\text{eval}}_c = \Delta_c \cdot \mathrm{Uniform}(0.8,\,2.2),\quad \text{seed}=999$$

3Digital twinnotebooks/train_twin.py · backend/digital_twin.py

Delta target

The twin predicts the change between the last timestep of the window and the next ground‑truth timestep:

$$\mathbf{y}^{(k)} = \mathbf{a}_{s_k+W}\; -\; \mathbf{a}_{s_k+W-1} \in \mathbb{R}^{F}$$

Control augmentation

With probability 0.35 per sample, a random control action is drawn and applied to the delta:

$$u^{\text{fuel}} \sim \mathrm{Uniform}(-0.12,\,0.12),\quad u^{\text{spark}} \sim \mathrm{Uniform}(-1.5,\,1.5)$$ $$\tilde{y}_{\,\text{Lambda}} = y_{\,\text{Lambda}} + 0.30\!\cdot\!u^{\text{fuel}},\quad \tilde{y}_{\,\text{IgnAngle}} = y_{\,\text{IgnAngle}} + 0.10\!\cdot\!u^{\text{spark}}$$

The two constants are stored as FUEL_TO_LAMBDA = 0.3 and SPARK_TO_IGN = 0.10. They encode the assumed physical gain of each control on its target channel.

Input tensor (window + repeated control row)

$$\tilde{X}^{(k)} = \bigl[\,X^{(k)} \,\|\, U^{(k)}\,\bigr] \in \mathbb{R}^{30\times(F+2)},\qquad U^{(k)}_{t,\cdot} = (u^{\text{fuel}}, u^{\text{spark}})\;\;\forall t$$

Architecture

$$ \begin{aligned} \mathbf{h}^{(1)} &= \mathrm{BN}\!\bigl(\mathrm{LSTM}_{128}(\tilde{X})\bigr)\in\mathbb{R}^{30\times 128},\;\;\text{Dropout}_{0.15}\\ \mathbf{h}^{(2)} &= \mathrm{BN}\!\bigl(\mathrm{LSTM}_{64}(\mathbf{h}^{(1)})\bigr)\in\mathbb{R}^{64},\;\;\text{Dropout}_{0.10}\\ \mathbf{u} &= \mathrm{ReLU}(W_1\mathbf{h}^{(2)} + b_1)\in\mathbb{R}^{64}\\ \hat{\mathbf{y}} &= W_2\mathbf{u} + b_2 \in \mathbb{R}^{F}\;\;(\text{linear}) \end{aligned} $$

Loss — mean squared error

With MAE as a side metric:

$$\mathcal{L} = \frac{1}{F}\sum_{f=0}^{F-1}(\hat{y}_f - y_f)^2,\qquad \mathrm{MAE} = \frac{1}{F}\sum_f |\hat{y}_f - y_f|$$
Optimizer & schedule
Adam · lr = 1e-3 · batch_size = 512 · epochs ≤ 50 · val_split = 0.12.

Next‑state reconstruction at inference

$$\hat{\mathbf{a}}_{\text{next}} = \mathbf{a}_{\text{last}} + \hat{\mathbf{y}}$$

Directional accuracy metric

$$\text{DAcc}_\lambda = \frac{1}{N}\sum_i \mathbb{1}\!\bigl[\,\mathrm{sign}(\hat{y}^{(i)}_\lambda) = \mathrm{sign}(y^{(i)}_\lambda)\bigr]$$

4SHAP explanationsnotebooks/run_shap.py

GradientExplainer with a 200‑sample background:

$$\phi_{f,c}(x) \approx \mathbb{E}_{x'\sim\mathcal{B}}\!\Bigl[\,(x_f - x'_f)\cdot \tfrac{\partial \hat{p}_c}{\partial x_f}\!\Bigr]$$

Per‑feature importance shown in the dashboard is the mean absolute SHAP across time and samples for the predicted class:

$$I_f = \frac{1}{NT}\sum_{i=1}^{N}\sum_{t=0}^{T-1}\bigl|\phi^{(i)}_{t,f,\,c^{*}_i}\bigr|$$

5Runtime use of the trained outputsbackend/classifier.py · backend/digital_twin.py · backend/simulation_engine.py

Confidence gating

$$c_{\text{gated}} = \begin{cases}c_{\text{raw}} & \text{if }\hat{p}_{c_{\text{raw}}} \ge t^{*}_{c_{\text{raw}}}\\0 & \text{otherwise}\end{cases}$$

Twin approval

$$\text{approve}\;=\;\bigl|\hat{\lambda}_{\text{next}} - \lambda_{\text{target}}\bigr| \;<\; \bigl|\lambda_{\text{current}} - \lambda_{\text{target}}\bigr|$$

with λ_target = 0 in z‑space.

Closed‑loop control step

For rich (Δ < 0) / lean (Δ > 0) faults:

$$u^{\text{fuel}}_k = -\mathrm{sign}(\Delta_k)\!\cdot\!\eta_{\text{fuel}},\qquad \Delta_{k+1} = \Delta_k - \mathrm{sign}(\Delta_k)\!\cdot\!\eta_{\text{fuel}}$$

For ignition fault:

$$u^{\text{spark}}_k = -\eta_{\text{spark}},\qquad \Delta_{k+1} = \max(0,\,\Delta_k - \eta_{\text{spark}})$$
Default step sizes
η_fuel = 0.02 · η_spark = 0.05

Convergence band

$$\bigl|\lambda_k - \lambda_{\text{target}}\bigr| < \text{LAMBDA\_BAND}$$

Summary

The two key learning losses are categorical cross‑entropy with label smoothing (classifier) and MSE on the next‑step delta (twin). Everything else — class weighting, threshold calibration, twin approval, control law — is post‑hoc machinery wrapped around those two losses.
